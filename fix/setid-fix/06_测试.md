# QQ Official `/setid` 真实测试规范

## 目的

本文件用于规范验证 `qq_official` C2C 私聊场景下 `/setid` 与宿主机稳定工作区入口是否按设计生效。

当前测试对象：

- 平台：`qq_official`
- 场景：`C2CMessage`
- 功能：`/setid`
- 宿主机稳定入口：`data/qq_workspaces/<appid>/<alias>/workspace`

## 测试前提

- 已部署并运行 AstrBot
- `sandbox` 已启用
- 实际 sandbox booter 为 legacy `shipyard`
- 当前测试用户使用同一个 QQ 机器人 C2C 私聊身份完成整套测试

## 每次测试前必须初始化

由于测试用户可能已经和机器人聊过天，因此**每轮测试前都必须恢复到可重复的初始化状态**，不能直接沿用上一次测试残留状态。

初始化完成的判定标准：

1. 宿主机不存在 `data/qq_workspaces/`
2. 宿主机不存在 `data/qq_workspaces/manifest.json`
3. `data/shipyard/bay_data/bay.db` 中：
   - `ships = 0`
   - `session_ships = 0`
4. `data/shipyard/ship_mnt_data/` 为空
5. `data/data_v4.db` 中：
   - 当前测试 QQ 会话不存在历史对话与 `sel_conv_id`
   - `platform_stats` 已清空（否则 Dashboard 的“总消息数/消息趋势分析”不会归零）
6. 如果启用了 Mnemosyne：
   - `data/plugin_data/astrbot_plugin_mnemosyne/message_counters.db` 中当前测试会话计数已清除
   - Milvus `default` 集合中当前测试会话的长期记忆已清除
7. `astrbot` 容器已重启，`6185` 可正常访问

## 初始化步骤

按顺序执行：

1. 删除宿主机稳定入口目录：

```bash
rm -rf /home/snight/AstrBot/data/qq_workspaces
```

2. 清空 legacy Shipyard 宿主机持久化目录：

```bash
rm -rf /home/snight/AstrBot/data/shipyard/ship_mnt_data/*
```

3. 清空 `bay.db` 中的 ship / session 记录：

```bash
python3 - <<'PY'
import sqlite3
from pathlib import Path

db = Path('/home/snight/AstrBot/data/shipyard/bay_data/bay.db')
conn = sqlite3.connect(db)
cur = conn.cursor()
cur.execute('DELETE FROM session_ships')
cur.execute('DELETE FROM ships')
conn.commit()
conn.close()
PY
```

4. 重启 AstrBot：

```bash
docker restart astrbot
```

5. 清空 AstrBot 主会话历史（否则不会等效 `/new`）：

```bash
docker exec astrbot sh -lc 'python3 - <<\"PY\"
import sqlite3
from pathlib import Path

user_id = "机器人1903:FriendMessage:<user_openid>"
db = Path("/AstrBot/data/data_v4.db")
conn = sqlite3.connect(db)
cur = conn.cursor()
cur.execute("DELETE FROM conversations WHERE user_id = ?", (user_id,))
cur.execute(
    "DELETE FROM preferences WHERE scope = ? AND scope_id = ? AND key = ?",
    ("umo", user_id, "sel_conv_id"),
)
cur.execute("DELETE FROM platform_message_history WHERE user_id = ?", (user_id,))
conn.commit()
conn.close()
PY'
```

6. 清空 Dashboard 全局消息统计（否则统计卡片不会归零）：

```bash
docker exec astrbot sh -lc 'python3 - <<\"PY\"
import sqlite3
from pathlib import Path

db = Path("/AstrBot/data/data_v4.db")
conn = sqlite3.connect(db)
cur = conn.cursor()
cur.execute("DELETE FROM platform_stats")
conn.commit()
conn.close()
PY'
```

7. 如果启用了 Mnemosyne，清空该测试会话的消息计数器：

```bash
docker exec astrbot sh -lc 'python3 - <<\"PY\"
import sqlite3
from pathlib import Path

session_id = "机器人1903:FriendMessage:<user_openid>"

```

8. 如果启用了 Mnemosyne，清空该测试会话在 Milvus 中的长期记忆：

```bash
docker exec astrbot sh -lc 'python3 - <<\"PY\"
from pymilvus import Collection, connections

session_id = "机器人1903:FriendMessage:<user_openid>"

    collection.delete(expr)
    collection.flush()
    print("deleted =", len(results))
PY'
```

9. 再次重启 AstrBot，清掉运行时缓存：

```bash
docker restart astrbot
```

10. 确认初始化状态成立：


## 真实测试步骤

以下步骤按顺序执行，不要跳步。

### 1. 验证首次私聊提示

在 QQ 机器人 C2C 私聊中发送一条普通消息，例如：

```text
你好
```

预期结果：

- 机器人返回一次 `/setid <3-10位小写字母或数字>` 提示
- 宿主机生成 `data/qq_workspaces/manifest.json`
- `manifest.json` 中 `alias_states[v1:<appid>:<user_openid>] = prompted`
- 此时还没有 alias 目录
- 此时还没有 `workspace` symlink

### 2. 验证 `/setid` 成功注册

在同一 C2C 私聊中执行：

```text
/setid tca001
```

预期结果：

- 机器人明确回复：
  - `ID 已注册成功`
  - `宿主机目录已创建`
  - `正式 workspace 已就绪` 或 `正式 workspace 尚未绑定，后续解析成功后会自动建立`
- 宿主机生成：
  - `data/qq_workspaces/<appid>/tca001/`
  - `data/qq_workspaces/<appid>/tca001/README.md`
- `manifest.json` 中生成：
  - `aliases["tca001"]`
  - `workspaces["v1:<appid>:<user_openid>"]`
  - `alias_states[...] = registered`

### 3. 验证首次 sandbox 绑定

继续在同一 C2C 私聊中触发一次真正会进入 legacy `shipyard` 的 sandbox 操作，例如 shell、文件工具或其他 sandbox tool。

预期结果：

- `bay.db` 中出现对应 `session_ships`
- `data/shipyard/ship_mnt_data/<ship_id>/...` 出现真实工作区
- 宿主机出现：

```text
data/qq_workspaces/<appid>/tca001/workspace
```

- `workspace` 必须是**相对 symlink**
- `manifest.json` 中对应 workspace 状态变为：
  - `status = resolved`
  - `reason = resolved_current`

### 4. 验证稳定入口可访问真实工作区

通过宿主机直接进入稳定入口目录，创建一个测试文件：

```bash
echo test-by-host > /home/snight/AstrBot/data/qq_workspaces/<appid>/tca001/workspace/host_check.txt
```

然后再通过 QQ 机器人让 sandbox 列目录或读取该文件。

预期结果：

- 宿主机和机器人看到的是同一份内容
- 说明稳定入口已正确指向真实 workspace

### 5. 验证 `/setid` 幂等

再次在同一会话执行：

```text
/setid tca001
```

预期结果：

- 返回幂等成功
- 不改变原有 alias 绑定关系
- 如果 `README.md`、`workspace` 或派生状态缺失，应被自动修复

### 6. 验证重启后持久化

执行：

```bash
docker restart astrbot
```

重启后，再次在同一 QQ C2C 私聊中触发 sandbox 操作。

预期结果：

- 仍落到同一 `workspace_identity`
- 稳定入口 `workspace` 仍可用
- 之前写入的 `host_check.txt` 仍存在

## 宿主机侧重点检查项

每轮测试至少检查以下三处：

1. 稳定入口：

```bash
ls -l /home/snight/AstrBot/data/qq_workspaces/<appid>/<alias>
```

2. 注册表：

```bash
cat /home/snight/AstrBot/data/qq_workspaces/manifest.json
```

3. legacy Shipyard 会话记录：

```bash
python3 - <<'PY'
import sqlite3
from pathlib import Path

db = Path('/home/snight/AstrBot/data/shipyard/bay_data/bay.db')
conn = sqlite3.connect(db)
cur = conn.cursor()
for row in cur.execute('select * from session_ships'):
    print(row)
conn.close()
PY
```

## 必须留存的证据

每次真实测试至少保留以下证据：

1. QQ 侧首次 `/setid` 提示截图
2. `/setid` 成功回复截图
3. 宿主机 `manifest.json`
4. `workspace` symlink 的 `ls -l` 输出
5. `bay.db` 中对应 `session_ships` 记录
6. 重启 AstrBot 后再次访问成功的证据

## 判定标准

满足以下条件才算通过：

- 首次私聊只提示一次 `/setid`
- `/setid` 成功后 alias 目录被创建
- 触发 sandbox 后稳定入口 `workspace` 被建立
- `workspace` 是相对 symlink
- 稳定入口能真实访问 sandbox workspace
- `/setid` 重复同值提交是幂等的
- AstrBot 重启后绑定关系和工作区内容保持不变

## 备注

- 当前用户即使“已经和机器人聊过天”，只要执行完上面的初始化步骤，仍可视为重新进入可重复的首次测试状态。
- 每轮测试前都必须重新初始化，避免上一次的 `manifest.json`、`bay.db`、`ship_mnt_data`、AstrBot 主会话历史或 Mnemosyne 长期记忆污染结果。
- 经证据确认，Dashboard 的“总消息数/消息趋势分析”来自 `data_v4.db.platform_stats` 全局聚合表；如果希望统计卡片也归零，必须一起清空它。
- 当前未发现 `platform_sessions`、`attachments`、`session_project_relations` 参与上述消息统计卡片取数，因此它们不属于本测试的必清项。
